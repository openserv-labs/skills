# OpenServ Workflow Architect - Platform Reference

Complete platform knowledge for generating deployment-ready workflow prompts. Every constraint in this file was discovered through production deployments and verified through multiple iterations.

---

## Project Structure

- Each agent MUST be its own directory with own `package.json`, `.env`, `tsconfig.json`, `.openserv.json`. Shared directories cause credential racing on startup.
- `"type": "module"` required in every `package.json`. Use ESM imports, never `require()`.
- Use `import dotenv from 'dotenv'; dotenv.config()` - NOT `import 'dotenv/config'` (can't reload after provision).

---

## Deploy Prerequisites (every agent directory)

**.gitignore** (required - package-lock.json causes 500 on upload):
```
node_modules/
package-lock.json
.openserv.json
.env
```

**.npmrc** (required - container npm install fails without):
```
legacy-peer-deps=true
```

**Peer deps** (undocumented but required):
```bash
npm install @privy-io/server-auth openai --legacy-peer-deps
```

**Deploy flow (per agent, in order):**

```bash
# Step 1: Provision - creates wallet, registers agent, writes .openserv.json
npm run dev
# Wait for "Connected" message, then Ctrl-C

# Step 2: Extract the auto-generated API key from .openserv.json into .env
node -e "const j=JSON.parse(require('fs').readFileSync('.openserv.json','utf8')); console.log('OPENSERV_USER_API_KEY='+j.apiKey)" >> .env

# Step 3: Deploy to OpenServ Cloud
npx @openserv-labs/client deploy .
```

**CRITICAL:** The `OPENSERV_USER_API_KEY` is NOT something the user has or gets from a dashboard. It is auto-generated by `provision()` and stored in `.openserv.json`. You MUST run `npm run dev` first to create it, then extract it into `.env` before deploying. The deploy CLI reads it from `.env`.

- If deploy fails with 500 or auth error, delete `OPENSERV_CONTAINER_ID` from .env and retry.
- Containers die randomly. Expect this. Delete container ID, redeploy.
- Container IDs change on redeploy but agent IDs do NOT.

## Provision + Auth

- Provision and deploy one agent at a time. Never simultaneously.
- `provision()` is idempotent for agent registration but creates a NEW workflow each time the definition changes.
- `provision()` writes `WALLET_PRIVATE_KEY`, `OPENSERV_API_KEY`, `OPENSERV_AUTH_TOKEN` to .env. Call `dotenv.config({ override: true })` after provision() to pick up new values.
- If .env doesn't end with a newline, appended vars concatenate. Always ensure trailing newline.
- Setup script needs its OWN independent wallet - NOT any agent's wallet. `PlatformClient.authenticate()` can invalidate agent API keys when using the same wallet.
- Two credential types: `OPENSERV_API_KEY` (agent, auto-managed) vs `OPENSERV_USER_API_KEY` (account, for deploy + PlatformClient). 401 = wrong key type.

## Model Parameters

Must include ALL THREE fields. API returns 400 without all three but doesn't say which is missing:

```typescript
model_parameters: {
  model: 'gpt-5',                // required
  verbosity: 'medium',           // required: 'low' | 'medium' | 'high'
  reasoning_effort: 'medium',    // required: 'low' | 'medium' | 'high'
}
```

- Set in `provision()`, NOT via `client.agents.update()` (returns 400).
- Run `client.models.list()` to discover available models.
- Do NOT use `claude-opus` models - platform orchestration overhead makes them 300s+ per task.
- Model choice affects the PLATFORM model's behavior, not just speed:
  - `claude-opus-4-6`: platform passes output through correctly (~153s)
  - `claude-sonnet-4-6`: platform model may rewrite entire output (~295s)
  - `gpt-5-mini`: may cause container 502, platform synthesizes manually (~534s)
  - `gpt-5`: works well for data tasks, reasonable speed

Recommended per role:
```
Resolver/data:      { model: 'gpt-5-mini', verbosity: 'low', reasoning_effort: 'low' }
Analysis/search:    { model: 'gpt-5', verbosity: 'medium', reasoning_effort: 'medium' }
Compiler/synthesis: { model: 'claude-opus-4-6', verbosity: 'medium', reasoning_effort: 'high' }
```

## Workflow Creation

- `workflows.create()` is the ONLY reliable way to wire trigger-to-task edges.
- **NEVER use `workflows.sync()` for trigger-wired workflows.** It creates task templates that never execute. Tasks stay "to-do" forever.
- Each `workflows.create()` call creates a NEW workflow with new trigger tokens.
- Workflow goal MUST be a detailed paragraph: what input it receives, what each stage does, what the output is. Vague goals get `INVALID_PROJECT_GOAL` rejection.
- Workflow name is user-facing (paywall page, marketplace). Polished and punchy - not kebab-case slugs.
- After creation: `client.triggers.activate()` each trigger AND `workflow.setRunning()`. Both required.
- After redeploying an agent to a new container, create a new workflow. Stale routing entries cause tasks to error.
- **No safe way to update a live workflow.** Every code change requires `workflows.create()` which produces new workflow ID + trigger tokens + paywall URL. sync() creates dead tasks, and redeploying agents breaks routing in old workflows. Accept this: build deployment scripts that output new URLs and update external references (reverse proxy, DNS redirect, or URL shortener in front of paywall).

## Edge Semantics

**Edges are mandatory.** The `dependencies` field on tasks does NOTHING. Only the `edges` array controls execution order. Every task must be reachable from a trigger via edges.

**Edges control TIMING, not data flow.** An edge A->B means "start B after A finishes." It does NOT pass A's output as B's input. B's platform model independently fetches A's output via `get-task-detail`. A task can access ANY task's data, even without a direct edge.

**Fan-OUT is AND. Fan-IN is OR.** This asymmetry causes most topology bugs.

- Fan-out: A->B + A->C starts B AND C in parallel. Correct.
- Fan-in: B->D + C->D starts D when EITHER finishes. D may run with incomplete data.

**Fix for fan-in:** ONE incoming edge from the SLOWEST predecessor. The final task fetches other data via `get-task-detail`.

```typescript
// WRONG: compile starts when EITHER intelligence or research finishes
edges: [
  { from: 'task:intelligence', to: 'task:compile' },
  { from: 'task:research', to: 'task:compile' },  // two incoming = OR
]

// RIGHT: compile starts only after research (the slower task)
edges: [
  { from: 'trigger:api', to: 'task:resolve' },
  { from: 'task:resolve', to: 'task:intelligence' },  // fan-out: parallel
  { from: 'task:resolve', to: 'task:research' },       // fan-out: parallel
  { from: 'task:research', to: 'task:compile' },        // fan-in: ONE edge
  // compile fetches intelligence output via get-task-detail, no edge needed
]
```

Write task bodies that name which tasks to fetch from: "Use the output from the resolve task and the intelligence task."

## The Platform Orchestration Model

Every task has a HIDDEN platform LLM between your trigger and your agent's capability. It:

1. Reads the task description to decide which capability to call and with what args
2. Passes the result to an "output reviewer" that scores it 1-10
3. Can REJECT output and retry (adding 30-60s per rejection)
4. Can REWRITE output instead of passing it through verbatim
5. Can STRIP fields from JSON output before storing the task result
6. Has workspace tools (file creation, todo lists) and USES THEM autonomously
7. May proceed without waiting for all data if it judges "good enough"

**Task body template (use for EVERY task):**
```
Call [capability_name] with [parameter descriptions referencing specific task outputs].
Return ONLY the raw JSON output from the capability - nothing else.
Do NOT create any files. Do NOT use todo lists.
```

## Capability run() Patterns

- **Type guard action before accessing task:** `if (action?.type === 'do-task' && action.task) { ... }`
- **Never use `this.process()`, `this.doTask()`, or `this.completeTask()`.** Just return from `run()`. The platform handles task completion.
- **Return value must be a string.** `return JSON.stringify(result)`

---

## Data Flow Between Tasks

- Platform wraps capability outputs. Downstream agents receive `{"inputs":{...}, "output":"..."}` not raw JSON. Use `extractPayload()` helper (see code patterns below).
- Text prepended before JSON gets stripped. Human-readable content must be INSIDE the JSON (`tldr` field).
- Only the FIRST task receives trigger input via `input: '{{trigger.field_name}}'`.
- Trigger input values are always STRINGS. Cast inside capability code.

---

## generate() Method

- Delegates LLM calls to the platform. No API key needed.
- `generate()` with `outputSchema` is BROKEN. Causes timeouts, empty responses, 502 errors. Never use it.
- Prompt for plain text JSON and parse with `match(/\{[\s\S]*\}/)`.
- When input data is empty, skip `generate()` entirely. Return structured defaults in code.

---

## Testing

- Always create a free webhook trigger ALONGSIDE x402 for testing.
- Webhook response only returns final result or timeout. Per-task errors visible only via `client.tasks.list({ workflowId })`.
- Fix broken agent -> redeploy THAT agent only -> re-fire webhook. Do NOT recreate workflow (unless agent moved to new container).
- x402 paywall may cache stale results. Use free webhook to confirm fixes.
- Containers die randomly. Delete `OPENSERV_CONTAINER_ID`, redeploy.

---

## Performance Budget

- Platform orchestration: 30-120s per task (includes output reviewer).
- 4-agent pipeline: 120-300s total.
- Parallelize independent agents via fan-out.
- Minimize `generate()` calls (15-60s each).
- Minimize search queries (2 max, 3 results each, 500 chars per result).
- Parallelize API calls within capabilities (`Promise.all`).
- Biggest speed win: task bodies the reviewer accepts on first pass.
- Timeouts: minimum 600s for webhooks/x402. 900s+ for complex pipelines.

## Build Sequence

**DO NOT** create an "orchestrator agent", "provisioner agent", or "provision-all" script. Each agent provisions itself independently. The setup script is a plain Node script in a `scripts/` directory, NOT an agent.

```bash
# For EACH agent, one at a time:
cd agent_dir
npm install --legacy-peer-deps
npm run dev              # provision + tunnel. Wait for "Connected".
# Ctrl-C

# Extract the auto-generated API key into .env:
node -e "const j=JSON.parse(require('fs').readFileSync('.openserv.json','utf8')); console.log('OPENSERV_USER_API_KEY='+j.apiKey)" >> .env

# Deploy to OpenServ Cloud:
npx @openserv-labs/client deploy .
cd ..

# After ALL agents deployed (not before):
cd scripts
# Generate independent wallet for setup script (NOT any agent's wallet)
node -e "console.log('WALLET_PRIVATE_KEY=0x' + require('crypto').randomBytes(32).toString('hex'))" > .env
npm install --legacy-peer-deps
npx tsx setup-workflow.ts    # Creates workflow with workflows.create()

# Test:
npx tsx test.ts
# If task errors: fix agent, redeploy, re-fire webhook
# Do NOT recreate workflow unless agent container changed
# Iterate until clean output
```

## Required Code Patterns

### extractPayload() - every downstream agent needs this

```typescript
function extractPayload<T>(raw: string, hasKey: string): T {
  let parsed: any
  try { parsed = JSON.parse(raw) } catch {
    const start = raw.indexOf('{')
    const end = raw.lastIndexOf('}')
    if (start >= 0 && end > start) parsed = JSON.parse(raw.slice(start, end + 1))
    else throw new Error(`Cannot parse: ${raw.slice(0, 200)}`)
  }
  if (parsed && typeof parsed === 'object' && hasKey in parsed) return parsed as T
  for (const field of ['output', 'result', 'data', 'value']) {
    if (parsed?.[field]) {
      const inner = typeof parsed[field] === 'string' ? JSON.parse(parsed[field]) : parsed[field]
      if (inner && hasKey in inner) return inner as T
    }
  }
  for (const val of Object.values(parsed || {})) {
    if (typeof val === 'string' && val.includes(hasKey)) {
      try { const nested = JSON.parse(val); if (nested && hasKey in nested) return nested as T } catch {}
    }
  }
  return parsed as T
}
```

### Cached API wrapper with backoff

```typescript
const cache = new Map<string, { data: any; expires: number }>()

function cached<T>(key: string, ttlMs: number, fn: () => Promise<T>): Promise<T> {
  const hit = cache.get(key)
  if (hit && hit.expires > Date.now()) return hit.data as Promise<T>
  return fn().then(data => { cache.set(key, { data, expires: Date.now() + ttlMs }); return data })
}

async function withBackoff<T>(fn: () => Promise<T>, retries = 3): Promise<T> {
  for (let i = 0; i < retries; i++) {
    try { return await fn() }
    catch (err: any) {
      if (err?.status === 429 && i < retries - 1) {
        await new Promise(r => setTimeout(r, 1000 * Math.pow(2, i)))
        continue
      }
      throw err
    }
  }
  throw new Error('Max retries')
}
```

### Safe generate() - never use outputSchema

```typescript
let result: any
try {
  const text = await this.generate({
    prompt: `Respond ONLY with a JSON object. No markdown, no code fences, no preamble.
Schema: ${JSON.stringify(schemaDesc)}
Data: ${JSON.stringify(data)}`,
    action
  })
  const match = text.match(/\{[\s\S]*\}/)
  result = match ? JSON.parse(match[0]) : fallbackDefault
} catch {
  result = fallbackDefault
}
```

## Production Deployment

- Set `DISABLE_TUNNEL=true` in .env for deployed agents. Container runs HTTP server only, no WebSocket tunnel. Platform reaches agents directly at `endpointUrl`.
- Set `FORCE_TUNNEL=true` to force tunnel mode even when `endpointUrl` is configured (useful for debugging deployed agents locally).
- Body parser limit is 10MB. If capability returns exceed this, chunk the output or write to workspace file and return a reference.

## Secrets Management

Never hardcode API keys in deployed agent code. Use OpenServ workspace secrets:

```typescript
// List all secrets available to the agent
const secrets = await agent.getSecrets({ workspaceId: action.workspace.id })
// Returns: { id: number, name: string }[]

// Get a specific secret value
const apiKeySecret = secrets.find(s => s.name === 'DOME_API_KEY')
if (apiKeySecret) {
  const apiKey = await agent.getSecretValue({
    workspaceId: action.workspace.id,
    secretId: apiKeySecret.id
  })
}
```

Set secrets via the platform dashboard. Access at runtime inside capability `run()` functions.

## Programmatic Discovery + Invocation

Other agents can find and pay for your x402 service without any human:

```typescript
// Discover all x402 services (no auth needed)
const services = await client.payments.discoverServices()
// Returns: { name, price, webhookUrl }[]

// Pay and invoke
const result = await client.payments.payWorkflow({
  workflowId: targetWorkflowId,
  input: { prompt: 'analyze this market' }
})
// Or by URL from discovery:
const result = await client.payments.payWorkflow({
  triggerUrl: services[0].webhookUrl,
  input: { ... }
})
```

x402 wallet payout address fallback chain: (1) explicit `walletAddress` on trigger config, (2) `client.walletAddress` from `authenticate()`, (3) derived from `WALLET_PRIVATE_KEY` env var.
